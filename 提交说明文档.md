# 中文场景文字识别比赛 - 提交格式与报错分析说明文档

## 1. 成功通过评测的标准格式
经过多次测试和第一性原理排查，比赛评测系统（后端脚本）对提交文件的格式有着极为严格的具体要求。最终成功通过评测的格式要素如下：

- **文件后缀**：必须是 `.txt` 文本文件（禁止直接提交 `.csv`）。
- **分隔符**：字段之间必须使用制表符 `\t` 分割（禁止使用逗号 `,`）。
- **表头名称**：第一列必须为 `new_name`，第二列必须为 `value`。
- **编码格式**：推荐使用标准 `UTF-8` 编码。

### 标准文件示例
```text
new_name	value
0.jpg	文本0
1.jpg	文本1
2.jpg	文本2
```

## 2. 之前提交失败原因（第一性原理分析）
比赛评测平台的后端评测代码存在硬编码和脆弱的解析逻辑，具体表现为：

1. **`KeyError: 'new_name'` 报错原因**：
   官方下发的训练集标注文件（`train_label.csv`）表头明明是 `name` 和 `value`。但评测机器人的后台脚本由于某种原因（可能是赛题组织方内部遗留逻辑），**强制要求测试集的表头必须叫 `new_name`**。当我们输出与官方训练集一致的 `name` 时，它用 pandas 读取并寻找 `df['new_name']`，自然触发了 `KeyError`。

2. **分隔符导致的列错位**：
   预测出的文本结果本身常常包含半角逗号（如 `Shanghai, Ltd`）。如果生成普通逗号分隔的 CSV，当平台后台解析文本时，带有逗号的预测结果会被强行切分成两列。这导致原本的 2 列变成了 3 列甚至更多，引发格式混乱，加剧了后台定位 `new_name` 及其对应值的失败。

3. **`0xb0` 解码报错原因**：
   虽然官方训练集数据是 `GBK` 编码，但是评测服务器运行的 Python 环境是默认将提交文件当作 `UTF-8` 来解析的。所以在尝试把结果转成 GBK 提交时，平台触发了 `UnicodeDecodeError`。

## 3. 终极生成代码 (转换流程)
为了之后免去手动麻烦，你可以直接将原始识别结果通过以下 Python 代码处理成 100% 稳妥的提交通道文件：

```python
import pandas as pd

# 读取我们最初生成的包含逗号分隔的基础预测数据
df = pd.read_csv('result.csv')

# 重构列名，满足平台硬性 'new_name' 要求
new_df = pd.DataFrame({
    'new_name': df['name'], 
    'value': df['value']
})

# 导出为平台指定的 \t 制表符 txt，无视文本内部残留的逗号
new_df.to_csv('result.txt', sep='\t', index=False, encoding='utf-8')
print("✅ 生成完毕：result.txt")
```
