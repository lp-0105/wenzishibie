# 实验报告 - 2026年4月14日

## 1. 实验背景
之前在云端 RTX 5090（Blackwell架构）配置环境中遭遇严重的 CUDA 209 核心兼容性报错，经过排查发现由于环境库不兼容无法强行跑通。为保证进度，训练策略全面转向**本地“无人房间”长时挂机模式**。
本次实验利用本地 8GB 显存设备，基于 TPS (Thin Plate Spline) 校正模块 + 双头网络（CTC + Attention），完成了从 100 轮到 200 轮的后半程极限冲刺训练。

## 2. 核心架构与训练策略
针对本地 8GB 显卡的硬件限制，对训练脚本 `train_scratch.py` 进行了以下深度魔改与压榨：
* **输入尺寸重构 ($48 \times 160 \rightarrow 32 \times 320$)**：
  * **高度 $48$ 降维至 $32$**：$32$ 像素高度是序列文字识别（如 CRNN/Transformer）的最佳“黄金标准”。经过主干网络（CNN）的下采样池化，32像素的图像高度会被极其完美地压缩成高度为 $1$ 的 1D 序列。这避免了因除不尽导致的特征错位，使得提取出的序列信息更加纯粹。
  * **宽度 $160$ 暴增至 $320$**：从根本上解除了“多字长文本”识别的封印。若保持原来的 160 宽度，长句子图片被暴力 resize 时，文字会被横向严重挤压，笔画粘连成“细面条”无法辨认；翻倍到 320 后，长句图片能够极大地保留原本正常的字符长宽比例（Aspect Ratio），显著提升了长句读取的鲁棒性。
* **快速收敛与大 Batch 策略（350轮 $\rightarrow$ 200轮）**：
  * **等效大 Batch 带来的方向确信度**：原本模型需要 350 轮才能磕磕绊绊收敛，现在砍到 200 轮就到达了极值。这得益于 `Batch_Size=100` 与 `ACCUM_STEPS=2`（梯度累加）的黄金组合。等效 Batch_Size 达到 200 后，每一次权重更新采样的图片样本极为丰富，极大程度抹平了单张异常图片的“噪音梯度”。优化器（AdamW）往正确谷底下降的“方向确信度”极高，几乎不走弯路，导致收敛速度大幅提前。
  * **大 Batch 赋能大步长（Learning Rate）**：只有巨大的等效 Batch Size，才兜得住我们设定的 `0.0005` 这么大的初始学习率。大方向准 + 大步子跑，这是能在区区不到两百轮内完成模型从初具雏形到精雕细琢的核心原因。
* **显存与硬件压榨**：
  * 物理 Batch Size 锁定为 100，避免 8GB 显存超载导致 OOM（最终平稳跑在 7470MB 左右）。
  * 梯度累加步数 (`ACCUM_STEPS`) 设为 2，实现等效 Batch Size = 200，保证梯度下降方向的稳定性。
  * 关闭之前所有的“养生降温”休眠代码 (`time.sleep`)。
  * `num_workers` 拉满到 8 并开启共享内存 (`use_shared_memory=True`)，保持 GPU 利用率长时间维持在 98% 满载状态。
* **分段数据增强 (Multi-stage Training)**：
  * **前半程 (Epoch 100-159)**：强开启旋转、空间透视扰动、噪点、对比度模糊等高强度数据增强（“苦练阶段”）。
  * **冲刺期 (Epoch 160-199)**：彻底关闭数据增强，在干净原始图上极致收敛损失函数。
* **学习率修复与退火**：
  * 修复了梯度累加未参与 `T_max` 计算导致的算不准问题，让 `CosineAnnealingDecay` 完美过渡。起点学习率从 ~0.00028 平滑回落至大结局的 0.000003，让末期的精微特征得以充分利用。
* **联合损失规划**：
  * CTC 损失权重 1.0 为主，Attention 损失权重 0.1 为辅（结合 0.1 的标签平滑 `label_smoothing` 防过拟合）。

## 3. 实验过程记录
* **前期踩坑（惨痛教训）**：因为最初的学习率衰减调度器（CosineAnnealingDecay）中 `T_max` 的计算没有将总步数除以梯度累加（`ACCUM_STEPS`），导致之前白跑了 200 轮。模型在 Epoch 199 时学习率仍然高达 0.00033，由于粗糙的学习率无法实现精细收敛，导致那时准确率死死卡在 46% 无法提升。
* **恢复阶段**：成功加载并衔接第 100 轮之前的底子，修复覆盖 BUG 并引入了按轮数备份的防覆盖机制（每 20 轮永久保存）。
* **进度预估修正**：修复了验证集时间带来的假性“每轮初耗时暴涨”错觉，利用全局累加次数算出了丝滑精确的 ETA。
* **冲刺阶段表现**：在 160 轮去掉数据增强后，验证集准确率稳步拉升到 48% 以上。

## 4. 最终结果
* **最佳验证集准确率 (Best Val Acc)**：**49.12%** （在第 190~195 轮期间触达顶峰并持续守擂成功）。
* **收官轮次准确率 (Epoch 199 Val Acc)**：49.08%。
* **算力消耗**：完美跑穿 200 轮，未发生高温降频或内存崩溃，证明本地硬件压榨策略绝佳。
* **线上榜单成绩 (Leaderboard)**：**62分**。相较于之前未使用 TPS 架构的 61 分模型，成绩有提升，但由于提升幅度仅为 +1 分，进步并不显著。

## 5. 分析结论与下一步计划
经过本地的精细调教，这套 TPS 纠偏双头模型已经完全吃透了现有训练数据，到达了局部最优解。但最终线上只提升了区区 1 分，我们进行如下拔高分析：

### 【为什么 TPS 带来的提升有限？】
1. **测试集分布倒逼**：增加 TPS （ Thin Plate Spline 薄板样条插值）的核心目的是“把弯曲、扭曲的字拉直”。如果测试集（赛题数据）中严重弯曲的形变图片比例本身就不大（大部分都是平行的手写或印刷体），那么 TPS 这个“屠龙刀”就缺乏用武之地，反而增加了网络训练的冗余结构负担。
2. **数据规模瓶颈**：当前模型容量已经触及了目前这批训练集的拟合上限卡脖子期。依靠在现有图像上做数据增强（如随机噪点、轻微旋转）虽然能防止过拟合，但已经不能塞给模型更多本质上的“新知识字符变体”了。

### 【下一步的突围方向】
要从 62 分继续往上硬拔，单纯死磕“炼丹调参（改 Batch Size / 调学习率 / 改迭代数）”收益已经极低，应当考虑如下真正影响识别率的大炮：
1. **针对性错误分析 (Bad Case Analysis)**：必须写一个小脚本，看看验证集里剩下的 50% 究竟识别错了什么？是字库里压根没有生僻字？还是繁体/简体混淆？还是长短句漏字？
2. **后处理与纠错 (Post-Processing / LM)**：目前采用的是纯视觉 CTC Greedy 贪心解码，非常容易出现同音错字（如通过视觉特征把“已”认成“己”）。若引入简单的语言模型 (N-Gram / 字典纠错解码)，可以无痛白嫖至少 3-5 分。
3. **合成数据降维打击**：利用 TextRender 工具，根据错例字生成海量的“人造手写干扰图”，把训练集规模扩大三倍喂给现有模型跑。
